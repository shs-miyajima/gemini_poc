# テスト計画レビューチェックリスト（AI 独立レビュー）— {{display_name}}

> 目的: `03-test-plan.md` §3 の「網羅基準」は**作成者（AI）のセルフチェック**であり、
> 作成者自身の見落としには気づけない。作成コンテキストを引き継がない**専用レビュー
> サブエージェント**（`sdd-plan-reviewer`、readonly）が、このチェックリストに沿って別視点で検証する。
>
> 運用: `.cursor/rules/sdd-workflow.mdc`「フェーズ 3 > 独立レビュー（承認確認前に必須）」を参照。
> このレビューは人間の承認を代替しない（自己承認の禁止の原則は変わらない）。

## 使い方

1. `03-test-plan.md` と各 CSV の作成後、**承認確認を人間に提示する前**に、専用レビューサブエージェント
   `sdd-plan-reviewer`（定義: `.cursor/agents/sdd-plan-reviewer.md`、readonly）へ
   `01-requirements.md` / `02-design.md` / `03-test-plan.md` / 各 CSV / 本チェックリストのパスを渡し、
   下記項目を ✓ / ✕ と理由で判定させる
2. ✕ の指摘のうち機械的に自動修正可能なもの（カバレッジ表の空欄、曖昧な期待結果の言い換え、
   出典 ID の記載漏れ、CSV 粒度の分割漏れ、**ピラミッド違反** 等）はメインの AI がその場で修正する
3. 業務ルールの解釈・仕様追加の要否など判断が必要な指摘は自動修正せず、承認確認の提示に
   含めて人間に判断を仰ぐ
4. 判定結果と対応状況を「6. 総合判定」の実施記録に記入する

## 0. テストピラミッド（レイヤ分担）

正本: `.cursor/rules/testing-pyramid.mdc`

- [ ] `03-test-plan.md` §1（テストピラミッド）が記載されているか
- [ ] 複合 VAL 分解表（§1.3）があり、サブルールごとに Test ID または E2E-only 理由があるか
- [ ] 閾値チェック一覧（§1.4）があり、NG/OK ペアが PHPUnit または Vitest にあるか
- [ ] E2E の `inp` が **1 画面 1 スモーク**＋クリティカル異常に留まっているか（項目ごとの inp 量産がないか）
- [ ] FormRequest / HTTP 422 の VAL が PHPUnit（`layer:unit` / `layer:http`）に計画されているか
- [ ] `02-design.md` §6 の Service 検証分岐が PHPUnit に 1:1 で計画されているか
      （代表サンプリングのみで親 VAL をカバー済みにしていないか）
- [ ] §1.3 / §1.4 の表が正本どおりの列定義になっているか
- [ ] §1.5（E2E 棚卸し）があり、**維持** と判定した Test ID のみが `03-test-plan.csv` に存在するか
- [ ] 各 E2E がメインフロー / クリティカル異常 / 画面スモークのいずれかであり、§1.5 に理由があるか
- [ ] 下層と同観点の E2E 重複がないか（件数比が高くても、重複がなければ削減不要）
- [ ] E2E `auth` に HTTP 403/404 のみを確認するケースが残っていないか（PHPUnit HTTP へ移行すべきか）
- [ ] 絞り込み・集計の条件分岐が **1 条件 1 E2E** になっていないか（代表 1 件 + PHPUnit 網羅か）
- [ ] 類似ジャーニー（ロール違い登録・ナビ単体・重複 dsp）が統合または削減されているか

## 1. トレーサビリティ

- [ ] 要件カバレッジ表に `01-requirements.md` の UC / VAL（§7）/ AC（§11）/ NFR（§9）/
      画面項目（§5.2）/ 権限（§8）の全項目が記載されているか（記載漏れがないか）
- [ ] カバレッジ表の「対応 Test ID」が空欄の行がないか（あれば意図的な対象外か確認する）
- [ ] NFR-xx が「対応不可」の場合、理由と代替手段（PHPUnit・手動確認等）が明記されているか
- [ ] `02-design.md` §5 で「影響あり」とした `IMPACT-xx` に、`03-test-plan.md`「回帰確認」で
      対応する Test ID が割り当てられているか
- [ ] CSV「出典」列の要件 ID が `01-requirements.md` に実在するか（typo・ID 不一致がないか）
- [ ] 逆方向（Test ID → 要件 ID）にも意味不明な出典（存在しない ID、根拠のない自由記述）がないか

## 2. 期待結果の質

- [ ] 期待結果が具体的な値・文言・件数・画面状態で書かれているか（「正しく表示される」等の曖昧語がないか）
- [ ] エラーケースでエラーメッセージの文言まで固定されているか（該当する場合）
- [ ] 同じ観点のケース間で期待結果の記述粒度が大きくブレていないか

## 3. ケース粒度・独立性

- [ ] 1 行 1 観点になっているか（複数の条件変化を 1 ケースに詰め込んでいないか）
- [ ] 前提条件が単独で再現可能な記載か（他の Test ID の実行結果を暗黙に前提していないか）
- [ ] 複数ケースが同一データを書き換える場合、干渉しないようテストデータが分離されているか
- [ ] 実行順序への依存（他ケースを先に実行しないと成立しない）がないか
- [ ] 並列実行安全性: 共有データ（Seeder の初期データ等）を更新・削除するケースが、そのケース専用の
      レコードを新規作成してから操作しているか（E2E 並列実行下での競合対策）

## 4. 網羅性

- [ ] 正常系・異常系が対になっているか（片方だけのルールがないか）
- [ ] 境界値の両側（ちょうど / 超過）が確認されているか
- [ ] 全ロール × 全操作の権限パターンが確認されているか（**PHPUnit HTTP で網羅**。E2E で全組み合わせを書いていないか）
- [ ] 要件で許可されている操作・状態遷移が **全レイヤ合算で** 列挙されているか（E2E だけで全件書いていないか）
- [ ] 更新系がある場合、必須項目・任意項目それぞれの空入力ケースが全項目分あるか

## 5. 実装可能性（先読みリスク）

- [ ] Playwright で実際に自動化できる内容か（外部連携・非同期処理・日付依存・メール送信等の
      技術的な難所が計画時点で想定されているか）
- [ ] PHPUnit / Vitest で担保すると書かれている項目が、設計書（`02-design.md`）の該当クラス・
      関数と対応しているか（担保先が曖昧でないか）
- [ ] 固定時間の待機（`waitForTimeout` 等）に頼った、フレーキーになりやすい書き方を
      前提にしていないか（要素の状態変化・ネットワーク完了を待つ設計になっているか）

## 6. 総合判定

| 状態 | 条件 | 対応 |
|------|------|------|
| 指摘なし | **0〜5** のすべての項目が ✓ | そのまま承認確認を人間に提示する |
| 自動修正済み | ✕ はあったが機械的に修正できた | 修正内容を記載の上、承認確認を人間に提示する |
| 要判断 | 業務ルールの解釈等、自動修正できない ✕ が残る | 承認確認に残課題として明記し、人間に判断を仰ぐ（自己判断で解消したことにしない） |
