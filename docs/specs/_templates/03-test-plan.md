- [ ] CSV の各行が 1 ケース 1 観点になっていること（対象項目・条件・期待結果が異なる場合は分割）
- [ ] 業務ルール・入力検証・権限・状態判定について、正常に通る代表ケースと異常系を対で確認していること
- [ ] 要件で許可されている操作・状態遷移をすべて列挙し、操作ごとに個別ケースを作成していること（一部の操作だけ確認して残りを省略しない）
- [ ] 条件分岐の取り得る値（例: 制限あり・一致 / 制限あり・不一致 / 制限なし）ごとにケースを作成していること
- [ ] セキュリティ入力（`<script>` タグ等のスクリプト文字列を含む入力が、保存後に無害化されて表示されること。ユーザーが入力し画面に表示される全項目が対象）

- [ ] 独立性・再現性（下記「独立性・再現性の標準観点」）を満たしていること

### 期待結果の書き方（曖昧表現の排除）

期待結果は、実行者や時期によって判定がぶれないよう、**検証可能な具体的な値**で書く。
「正しく」「適切に」「問題なく」等、判定基準を含まない表現は使わない。

| NG 例 | OK 例 |
|--------|--------|
| 正しく表示される | 一覧に該当データが 5 件、氏名の昇順で表示される |
| 適切にエラーになる | 氏名欄の下に「氏名を入力してください」が表示され、保存されない（DB 件数が変化しない） |
| 問題なく保存される | 一覧画面に遷移し、登録した氏名・メールアドレスが 1 行目に表示される |

- バリデーションエラーは「エラーになること」だけでなく、**エラーメッセージの文言**まで固定する
- 画面表示は「表示される」だけでなく、**件数・並び順・表示位置**等、確認可能な粒度まで書く

### 独立性・再現性の標準観点

各テストケースは、他のケースの実行結果や実行順序に依存せず、**単独で実行しても成立する**ように書く。

- [ ] 前提条件は、そのケース単独で再現可能なデータ状態を明記している（他の Test ID が作成したデータを暗黙の前提にしない）
- [ ] 複数のケースが同一の共有データ（マスタ・既存レコード等）を書き換える場合、ケース間で干渉しないようテストデータを分離している（例: ケースごとに固有のレコードを新規作成してから使う）
- [ ] 実行順序を入れ替えても、または他のケースを実行せずに単体で実行しても、期待結果が成立する
- [ ] Seeder 等の共通初期データに依存する場合は、その旨を前提条件に明記している（「§4 テストデータ」の内容と矛盾しない）

#### 並列実行安全性（Playwright は `fullyParallel: true`）

E2E は複数ワーカーで同時に実行される前提で書く。「単独実行なら成立する」だけでは不十分で、
**他のケースと同時に実行されても干渉しない**ことを確認する。

- [ ] 一覧の 1 行目・件数等、Seeder の初期データをそのまま参照するケースは、他のケースがその
      データを更新・削除しないことを確認している（更新系のケースは別レコードを使う）
- [ ] 同一レコード（同一ユーザー、同一マスタ行等）を複数のケースが更新する場合、ケースごとに
      専用のレコードを新規作成してから操作する（既存レコードの奪い合いにしない）
- [ ] 採番・一意制約・排他ロックなど、並列実行で競合しうる処理を対象とするケースは、
      「同時実行・競合」の観点（上記）で個別に検討済みである

### CSV ケース粒度の標準観点

各 CSV は **1 行 1 観点** を原則とする。

| 分割が必要な例 | 分割方針 |
|----------------|----------|
| 複数の入力項目をまとめている | 入力項目ごとに別ケースにする |
| 複数のエラー理由をまとめている | エラー理由ごとに別ケースにする（例: 未指定、形式不正、存在なし） |
| 複数のロールや権限をまとめている | ロールや権限パターンごとに別ケースにする |
| 複数の状態・ステータスをまとめている | 期待結果や遷移が異なる場合は状態・ステータスごとに別ケースにする |

複数条件を 1 行にまとめるのは、同じ操作・同じ期待結果として同一の確認観点である場合に限る。

### 更新系の標準観点（編集機能がある場合は必須）

要件の画面項目定義（必須/任意）に照らして、**全項目**について以下を確認する。

| 項目種別 | 空にして更新した場合の期待動作 | テストケース |
|----------|-------------------------------|-------------|
| 必須項目 | バリデーションエラーとなり、DB は元の値のまま | 必ず作成 |
| 任意項目（空入力 OK） | **空値（null / 空文字）で上書き保存される**（「空なら元の値を維持」にしない） | 必ず作成 |

※「空なら元の値を維持する」仕様にしたい場合は、要件定義（01-requirements.md）に明記されていることを確認する。明記がなければ open-questions で確認する。
